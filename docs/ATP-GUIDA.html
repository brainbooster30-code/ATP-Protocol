ATP v1.8

Agent Transport Protocol · Protocollo crittografico peer-to-peer per agenti autonomi

1. Cos'è ATP

Il problema

Due agenti software autonomi — su due macchine diverse, forse su due continenti diversi — vogliono comunicare in modo sicuro. Non si fidano l'uno dell'altro a priori. Non condividono un server centrale, non hanno un'autorità di certificazione comune, e non possono assumere che la rete sia trusted.


ATP risolve questo problema fornendo un canale cifrato, verificabile e federato dove ogni agente possiede una identità autocertificata e ogni messaggio è firmato. Zero server centrale, zero intermediari, zero configurazione di rete.

Le parole chiave

  • MCC — Merkle-Claim Card: documento d'identità crittografico autocertificato. Come un passaporto, ma senza un governo che lo emette.
  • Handshake 5 fasi — TLS → Version → MCC → Proof-of-Possession → Capability. Due agenti si identificano, si verificano e concordano le capacità.
  • E2E Encryption — X25519 ECDH + AES-256-GCM + Ed25519 firma. Anche se il TLS viene violato, il payload rimane cifrato e autenticato.
  • CuckooFilter — Struttura dati per revoca distribuita. Un agente compromesso viene revocato e la revoca si propaga a tutti i peer via gossip.
  • RootStore — Registro distribuito di autorità fidate. Ogni agente ha un RootStore locale che viene sincronizzato con i peer.

2. A cosa serve

Cooperazione tra agenti AI

Due o più agenti AI collaborano su task complessi:

  • Un agente ricerca → un altro verifica i fatti → un terzo summarizza
  • Pipeline di code review: dev → review → fix
  • Voting distribuito tra 4+ agenti con attestazione crittografica

Automazione aziendale

Reparti diversi hanno agenti specializzati che comunicano direttamente:

  • R&D → Operations per dati di produzione
  • Security → HR per segnalazioni anonime
  • Finance → R&D per approvazione budget

Federazione di istanze

Più server ATP che formano una rete federata:

  • Scuola ↔ Insegnante ↔ Studente
  • Sedi aziendali distribuite
  • Ricercatori indipendenti che condividono capacità di calcolo

Sicurezza e audit

Ogni scambio è verificabile crittograficamente:

  • Firme Ed25519 su ogni frame
  • Revoca distribuita con gossip TCP
  • RootStore persistente per verifica post-eventum
  • Metriche Prometheus per monitoring

2.5. Perché ATP — confronto con i metodi tradizionali

Il problema dei vecchi approcci

Prima di ATP, far comunicare due agenti software significava scegliere un compromesso tra semplicità e sicurezza — e spesso si perdeva su entrambi.


HTTP/REST — Il più comune. Un agente chiama un'API, l'altro risponde. Sembra funzionare finché non ti accorgi che:

  • Serve un server centrale (o almeno un indirizzo IP pubblico)
  • L'autenticazione è demandata a JWT/API key — nessuna identità crittografica nativa
  • Il payload viaggia in chiaro se non c'è HTTPS, e anche con HTTPS non c'è E2E
  • Nessuna revoca nativa, nessun gossip, nessuna proof-of-possession
  • Ogni chiamata è stateless — niente multiplexing, niente streaming nativo

gRPC — HTTP/2, protobuf, bi-directional streaming. Tecnicamente superiore a REST, ma:

  • Dipende da HTTP/2 — non è un protocollo di trasporto, è un framework RPC
  • Nessuna identità autonoma: i peer non possono verificarsi senza un CA esterna
  • Nessuna revoca distribuita, nessun gossip
  • Complesso da configurare (protobuf, codegen, load balancer)

WebSocket — Full-duplex, ma:

  • Nessuna crittografia nativa (dipende da WSS = TLS)
  • Nessuna identità, nessuna firma, nessuna revoca
  • Nessun multiplexing: un solo canale, messaggi sequenziali
  • Nessun handshake crittografico: è solo un upgrade HTTP

MQTT — Leggero, pub/sub, ma:

  • Richiede un broker centrale — single point of failure
  • Nessuna identità peer-to-peer: l'autenticazione è username/password o certificato
  • Nessuna crittografia end-to-end (solo TLS broker-to-client)
  • Nessuna revoca distribuita, nessuna proof-of-possession

Cosa rende ATP diverso

ATP non è "un altro protocollo di trasporto". È un protocollo di identità e cooperazione progettato specificamente per agenti autonomi. Ogni decisione di design parte da una domanda: "Come fa un agente a fidarsi di un altro agente che non ha mai incontrato prima?"


Le differenze fondamentali:

  • Identità autocertificata — L'MCC è un documento firmato dall'agente stesso. Nessuna CA necessaria. Il peer verifica la firma, non chiede a un'autorità.
  • Trust senza server centrale — Due agenti possono stabilire una connessione sicura senza mai parlare con un terzo. Questo è impossibile con HTTP/gRPC/WebSocket.
  • Crittografia a strati — TLS (trasporto) + ECDH X25519 (E2E) + Ed25519 (firma). Anche se il TLS viene violato, il payload rimane cifrato e autenticato. Nessun altro protocollo offre questo out-of-the-box.
  • Revoca distribuita — CuckooFilter + gossip TCP. Un agente compromesso viene revocato e la revoca si propaga a tutti i peer. Nessun server centrale di revoca.
  • Niente codice finto — BLAKE3 obbligatorio, niente fallback, niente demo mode bypassabile. Ogni test è reale (86 test, 0 mock).

Quando NON usare ATP

ATP è progettato per comunicazione agente ↔ agente. Non è adatto per:

  • Browser ↔ Server — Usa HTTP/HTTPS. I browser non parlano CBOR nativamente.
  • Microservizi in stessa LAN — gRPC o messaggistica leggera sono più semplici.
  • Streaming video/audio in tempo reale — WebRTC è meglio per media in tempo reale.
  • IoT con vincoli estremi di memoria — MQTT o CoAP sono più leggeri.
  • Single-node / monolite — Se tutto gira sulla stessa macchina, chiamate dirette bastano.

La regola pratica: se hai bisogno che due agenti si identifichino, si verifichino e collaborino su internet senza un server centrale, ATP è la scelta giusta. Altrimenti, usa lo strumento più semplice che funziona.

3. Come funziona

Stack a 5 layer

┌────────────────────────────────────────────────────────────┐
│  TASK LAYER                                                 │
│  send_task('deepseek', payload) → risposta strutturata      │
│  Multiplexing per task_id (task concorrenti)                 │
│  Fire-and-forget (DeepSeek 60s non blocca il server)        │
│  Task streaming (partial=true, risposte parziali)           │
├────────────────────────────────────────────────────────────┤
│  E2E CRYPTO LAYER                                           │
│  X25519 ECDH → BLAKE3 KDF → AES-256-GCM cifratura payload  │
│  Encrypt-then-sign: Ed25519 firma sul ciphertext            │
│  Verify-then-decrypt: rifiuta se la firma è invalida        │
├────────────────────────────────────────────────────────────┤
│  CONTROL LAYER                                              │
│  SHUTDOWN / REVOKE_NOTIFY / HEALTH / PING+PONG              │
│  ROOT_STORE_UPDATE (chain-of-manifests firmati)             │
├────────────────────────────────────────────────────────────┤
│  HANDSHAKE LAYER (5 fasi)                                   │
│  TLS 1.3 mutual → Version → MCC exchange + PoP → Cap       │
│  HandshakeRateLimiter: 10 tentativi/60s per IP             │
├────────────────────────────────────────────────────────────┤
│  TRANSPORT LAYER (dual)                                     │
│  TCP + TLS 1.3 (CA Ed25519 condivisa)                      │
│  oppure QUIC (RFC 9000, aioquic, RSA 2048)                 │
└────────────────────────────────────────────────────────────┘

Handshake 5 fasi — passo passo

1. 🔐  TLS 1.3           Mutual TLS con CA condivisa
2. 📋  VERSION           Negozia versione protocollo
3. 🆔  MCC + PoP         Scambio identità + proof-of-possession
4. 🧠  CAPABILITY        Annuncio capacità (DeepSeek, max task, etc.)
5. ⚡  TASK              Inizio scambio task

Dopo handshake:
  • E2E key derivation: X25519 ECDH + BLAKE3 (sorted keys)
  • RootStore push: entrambi i lati condividono le proprie autorità
  • Keepalive PING/PONG ogni 30s

Revoca distribuita

1. Un agente viene compromesso → seriale revocato
2. Revoca inserita nel CuckooFilter locale
3. Gossip TCP (porta 8444) propaga la revoca ai peer
4. Ogni peer aggiorna il proprio CuckooFilter
5. Al prossimo handshake, l'agente revocato viene rifiutato

DegradationPolicy:
  CONFIRMED    → peer visto di recente, RootStore sincronizzato
  STALE        → peer non visto da > 1h, usare con cautela
  UNCERTAIN    → peer sconosciuto, handshake completo richiesto

4. Cosa ci si può fare — esempi reali

Due agenti che collaborano (TCP)

import asyncio
from atp_sdk import SimpleATPServer, SimpleATPClient

async def main():
    srv = SimpleATPServer(); await srv.start()
    cli = SimpleATPClient("ricercatore"); await cli.connect()
    
    risultato = await cli.chat(
        "Cerca le ultime scoperte sulle reti neurali"
    )
    print(risultato)
    
asyncio.run(main())

Stessa cosa, via QUIC (RFC 9000)

import asyncio
from atp_quic import QUICServer, QUICClient

async def main():
    srv = QUICServer(); await srv.start('127.0.0.1', 18900)
    cli = QUICClient(); await cli.connect('127.0.0.1', 18900)
    
    result = await cli.send_task('echo', 'Ciao ATP!')
    print(result['status'])  # 'ok'
    
asyncio.run(main())

Pipeline a 3 agenti

# research_assistant.py — 3 agenti in pipeline
1. Researcher   → cerca informazioni su DeepSeek
2. FactChecker  → verifica le fonti
3. Summarizer   → produce un report finale

Ogni agente è un SimpleATPServer con un task_handler specializzato.
I task vengono inoltrati da un agente all'altro via ATP.
